# Multi-Angle Review

このスキルは**複数サブエージェントによる多角的コードレビューをバグ0までループ実行する**スキルです。

単一ファイル内のロジックだけでなく、**ファイルの前後関係・ファイル間の影響度**まで多層的にレビューし、既存機能へのデグレを防ぎながらバグを根絶します。言語非依存で、規模に応じてサブエージェント数を可変調整します。

---

## 🎯 このスキルの目的

1. **多角的レビューでバグ見逃しを最小化**
   - 複数サブエージェントを並行起動し、異なる観点から同時レビュー
   - 1エージェントの盲点を他エージェントがカバー

2. **3階層のレビューで既存機能のデグレを防止**
   - Layer 1: 単一ファイル内ロジック
   - Layer 2: ファイル前後関係（関数・変数の呼び出しチェーン）
   - Layer 3: ファイル間影響度（import先・共有構造・テスト）

3. **バグ0までループ実行**
   - 修正 → 再レビュー のループを、バグ0 or ユーザー停止まで継続
   - 修正が新たなバグを生んでいないか毎ラウンド検証

4. **言語非依存**
   - Python / JavaScript / TypeScript / Go / Rust / Ruby / Java / C# / PHP / SCSS / EJS など
   - レビュー観点を普遍的概念で定義し、言語固有のベストプラクティスはサブエージェント側が適用

5. **ソースコード編集は必ずユーザー承認制**
   - 各ラウンド後にバグ一覧を報告 → ユーザー承認 → 修正実行
   - ユーザーの意図しない変更を防ぐ

## 🚀 使い方

### 発動トリガー

以下のようなユーザー発言でこのスキルを発動します：

- 「このコードをレビューして」
- 「バグがないかチェックして」
- 「多角的にレビューして」
- 「バグ0までレビュー回して」
- 「前後関係・ファイル間関係性も含めてレビューして」

### 基本的な呼び出し方

```

{対象ファイル} を多角的にレビューしてバグ0まで回してください。
```

## 🎚️ レビュー深度の確認（発動直後・必須）

スキル発動時、Phase 0 より前に**ユーザーにレビュー深度を確認する**。

### 確認プロンプトフォーマット

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎚️ レビュー深度を選択してください

対象: {対象ファイル / ディレクトリ}

【深度オプション】

A) Layer 1 のみ（単一ファイル内レビュー）
   - 対象: 1ファイルのロジック・境界条件・エラーハンドリング
   - 適用ケース: 独立したユーティリティ関数、単発スクリプト
   - 速度: 最速

B) Layer 1 + Layer 2（前後関係まで）
   - 対象: 上記 + 関数間呼び出しチェーン・後方互換性
   - 適用ケース: 複数関数を持つモジュール、クラス単位のレビュー
   - 速度: 中

C) Layer 1 + Layer 2 + Layer 3（全階層・推奨）
   - 対象: 上記 + ファイル間影響度・共有型・公開API契約
   - 適用ケース: アプリケーション、複数ファイル連携、プロダクション品質
   - 速度: 最も時間がかかる

D) 自動判定
   - Claudeが規模判定スコアから最適な深度を提案
   - 小規模: A、中規模: B、大規模以上: C を基本とする
   - 提案後にユーザー確認を求める

どの深度でレビューしますか？（A/B/C/D）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 深度別のフロー調整

| 深度 | Phase 0調査深度 | 起動Agent |
|------|----------------|----------|
| **A (Layer 1のみ)** | 最小（対象ファイルのみ・依存調査不要） | Agent A（Layer1）+ Agent D（Tests）の1-2体 |
| **B (Layer 1+2)** | 小～中（同ファイル内呼び出し・直接import把握） | Agent A + B + D の2-3体 |
| **C (全階層)** | 規模判定スコアに応じた深度（小/中/大/超大） | Agent A + B + C + D の3-5体 |
| **D (自動)** | Claudeが判断 | 規模に応じて可変 |

### 深度選択の判断基準（ユーザー向けガイド）

- **単発スクリプト・独立関数** → A
- **クラス1つ・モジュール1つ** → B
- **複数ファイル連携・共有型あり** → C
- **アプリケーション全体** → C
- **迷ったら** → D（自動判定）

### 深度変更の柔軟性

- レビュー途中で「やっぱり深く見て欲しい」と言われたら、次ラウンドから深度を上げる
- 逆に「もっと軽く」と言われたら、次ラウンドから深度を下げる
- 深度変更時はユーザーに再確認する

## 📐 レビュー3階層の観点定義（核心）

### Layer 1: 単一ファイル内レビュー

**観点:**
- ロジック正しさ（アルゴリズム・計算式）
- 境界条件（空配列・null/undefined・負の値・0・最大値）
- 型の整合性
- エラーハンドリング（例外処理・リソース解放）
- 同一ファイル内の関数間呼び出し
- デッドコード（未使用変数・到達不能コード）
- 重複ロジック

**典型的な発見例:**
- off-by-one エラー
- null参照
- 無限ループの可能性
- 未catchの例外

---

### Layer 2: ファイル前後関係レビュー ⚡重要

**観点:**
- 関数Aの修正が関数B（同ファイル・呼び出し元）に与える影響
- 変数・状態の伝播経路
- 修正前に通っていた分岐が修正後も通るか（後方互換性）
- 副作用の変化（グローバル変数・ファイルI/O・DB更新）
- 関数シグネチャ変更の波及

**典型的な発見例:**
- 戻り値の型変更により呼び出し元が壊れる
- 新規追加した早期returnにより後続処理がスキップされる
- 状態変数の初期化タイミング変更による挙動変化

---

### Layer 3: ファイル間影響度レビュー ⚡重要

**観点:**
- モジュールA修正がモジュールB（import先・呼び出し元）に与える影響
- 共有データ構造（型・スキーマ・定数・Enum）の一貫性
- テストファイルへの影響
- 設定ファイル・環境変数との整合性
- 公開API（外部から呼ばれる関数）の契約維持
- DIコンテナ・イベントバス経由の間接依存

**典型的な発見例:**
- 関数削除により依存モジュールがimportエラー
- 共有型の定義変更により別ファイルの型エラー
- 定数値変更により別ファイルのテストが失敗
- 公開APIのシグネチャ変更による外部契約違反

## 🔢 サブエージェント数の判定基準

対象コードの**規模と複雑度**を判定し、サブエージェント数を決定します。

### 判定テーブル

| 規模 | 判定基準 | エージェント数 | 役割分担 |
|------|---------|--------------|---------|
| **小** | 1ファイル・300行未満・依存少 | **1-2体** | ロジック+境界条件 / テストカバレッジ |
| **中** | 1-3ファイル・1000行未満・依存中 | **3体** | Layer1 / Layer2 / Layer3+テスト |
| **大** | 4ファイル以上 or 1000行以上・依存多 | **3-5体** | ファイル毎Layer1 / Layer2統合 / Layer3統合 / テスト |
| **超大** | モジュール横断・3000行以上・依存複雑 | **5体以上** | ファイル群毎 / 関係性 / データフロー / 契約 / テスト |

### 規模判定ロジック

```
スコア = ファイル数 × 10 + 総行数/100 + 依存ファイル数 × 5

- 小: スコア 〜30
- 中: スコア 30-80
- 大: スコア 80-200
- 超大: スコア 200〜
```

依存ファイル数は `Grep` で import/require/include 文をカウントして算出。

### 観点ごとのエージェント割り当て（中〜大規模の例）

```
Agent A: Layer 1 - ロジック・境界条件・エラーハンドリング
Agent B: Layer 2 - 関数前後関係・呼び出しチェーン・状態伝播
Agent C: Layer 3 - import依存・共有構造・公開API契約
Agent D: テストカバレッジ分析（既存テストが変更をカバーできているか）
Agent E: （超大のみ）データフロー全体・副作用の追跡
```

---

## 📂 大規模ファイルの分割戦略

### サイズ別分割方針

| ファイルサイズ | 分割戦略 |
|--------------|---------|
| **〜500行** | 分割なし、1エージェントで読み切り |
| **500-1500行** | 機能単位で2-3セクション分割、各セクションを並行レビュー |
| **1500-3000行** | クラス/モジュール単位で分割、統合レビュー担当を1体追加 |
| **3000行以上** | 事前にGrep/Globで構造把握、優先度順にレビュー（変更箇所＋依存箇所のみ） |

### 分割時の注意事項

- 分割境界で文脈が途切れないよう、各エージェントに以下を渡す：
  - 担当セクションの前後100行
  - ファイル全体の構造概要（関数・クラス一覧）
  - 依存関係マップ（import/呼び出し元リスト）
- **統合レビュー担当を1体追加**し、分割間の整合性をチェック

### 各フェーズの詳細

#### Phase -1: レビュー深度確認 ⚡必須

スキル発動直後、**何よりも先に**ユーザーにレビュー深度を確認する（上記「🎚️ レビュー深度の確認」セクション参照）。

- ユーザーが深度を指定していない場合は必ず確認プロンプトを出す
- 深度が決まるまでPhase 0に進まない
- ユーザーが最初から「Layer 1だけでいい」「全階層で」と指定していれば、その深度を使う

#### Phase 0: アーキテクチャ調査 ⚡重要（深度Cの場合）

Layer 3（ファイル間影響度）を表層的で終わらせないために、**事前にプロジェクト全体のアーキテクチャを調査**し、全エージェントが参照する「アーキテクチャマップ」を作成する。

##### 深度別のPhase 0実行内容

| 深度 | Phase 0の実行内容 |
|------|-----------------|
| **A (Layer 1のみ)** | 対象ファイルのサイズ・行数把握のみ。アーキテクチャ調査スキップ |
| **B (Layer 1+2)** | 上記 + 同ファイル内の関数一覧・直接import把握 |
| **C (全階層)** | **下記の全調査ステップを規模別深度で実行**（小/中/大/超大） |
| **D (自動)** | 規模判定後に深度を決定し、対応する調査を実行 |

##### 調査の基本ステップ（深度C または D選択後）

1. 対象ファイルを特定（ユーザー指定 or 直近編集ファイル）
2. `Glob` / `Read` で関連ファイル・行数を把握
3. `Grep` で import/require/include をカウント
4. 規模判定ロジックでスコア算出
5. **規模に応じた深度でアーキテクチャ調査を実行**（下記テーブル参照）
6. **アーキテクチャマップを成果物として作成**
7. サブエージェント数・役割決定
8. ユーザーに準備結果を報告

##### 規模別の調査深度テーブル

| 規模 | 調査内容 |
|------|---------|
| **小** | ・直接import/require一覧<br>・ファイル数・行数把握 |
| **中** | 小の内容 +<br>・依存グラフ2層展開（import先のimport先まで追跡）<br>・共有モジュール特定（types/, constants/, utils/, config/等）<br>・設定ファイル検出（.env, config.*, settings.*）<br>・対応テストファイル配置確認 |
| **大** | 中の内容 +<br>・アーキテクチャパターン判定（MVC/レイヤード/モジュラー/フラット）<br>・公開API境界特定（index.*, __init__.py, mod.rs等のエクスポート集約点）<br>・間接依存の追跡（3層先まで）<br>・共有状態の検出（グローバル変数・シングルトン） |
| **超大** | 大の内容 +<br>・**専用「アーキテクチャマッパー」エージェント起動**<br>・アーキテクチャマップ成果物をファイル化<br>・イベントバス・DIコンテナ経由の間接依存追跡<br>・設定ファイル経由の文字列参照（関数名・クラス名）検索 |

##### 「アーキテクチャマッパー」エージェント（超大規模のみ）

超大規模PJでは、Phase 0専用のサブエージェントを1体起動し、以下を実行：

- プロジェクトルートから `Glob` で全ソースファイルツリー取得
- ディレクトリ構成からアーキテクチャパターン推定
- 主要エントリーポイント特定
- 共有モジュール・公開API境界の網羅的抽出
- 間接依存の追跡（イベントバス・DI・設定経由の文字列参照）
- アーキテクチャマップを成果物として出力

この成果物が、Layer 2/3担当エージェントの精度を大きく左右する。

#### Phase 1: 並行レビュー

- `Agent` ツールで複数サブエージェントを**1メッセージ内で並行起動**
- 各エージェントに以下を渡す：
  - 対象ファイルパス
  - レビュー観点（Layer 1/2/3のどれか）
  - **Phase 0で作成したアーキテクチャマップ**（全エージェント共有）
  - 前回ラウンドの修正内容（ラウンド2以降）

**特にLayer 2/3担当エージェントは、アーキテクチャマップを起点に影響範囲を判断する。**

#### Phase 2: 統合・報告

- 各エージェントの報告を統合
- 重複バグを除去（同じ箇所を複数エージェントが指摘した場合は1件に）
- HIGH/MEDIUM/LOWで分類（基準は後述）
- ユーザーに報告

#### Phase 3: ユーザー承認

- ユーザーが修正対象を選択（全件 / 一部 / 却下）
- 却下されたバグは次ラウンドで再評価対象外

#### Phase 4: 修正実行

- 承認されたバグのみ `Edit` / `Write` で修正
- テスト存在時は実行し、結果を報告
- 既存テストが壊れた場合は即座にユーザーに報告

## ✅ バグ分類基準

### HIGH（致命的）

- データ破損・データ損失を引き起こす
- セキュリティリスク（認証バイパス、SQL injection等）
- 計算結果を構造的に歪めるバグ（look-ahead bias等）
- 無限ループ・クラッシュ
- 公開API契約違反

### MEDIUM（重要）

- 特定条件下での誤動作
- エラーハンドリング不備
- 後方互換性の破壊（影響範囲限定）
- パフォーマンス重大劣化
- テストでカバーされていない主要分岐

### LOW（軽微）

- デッドコード・未使用変数
- 命名の一貫性欠如
- 冗長なロジック
- コメント不足
- マイクロパフォーマンス

## ⚠️ 重要な注意事項

### 1. ソースコード編集は必ずユーザー承認制

- 検出したバグを**独断で修正しない**
- 必ずバグ一覧を報告 → ユーザー承認 → 修正実行の順で進行
- 却下されたバグは次ラウンド以降も再提案しない

### 2. テストファイル・ドキュメントも同様

- テストコードの修正も承認対象
- バグ修正に伴うテスト変更は、「修正案」として一緒に提示

### 3. 修正が新たなバグを生むリスク

- ラウンドN+1では必ず**N で修正した箇所周辺**を重点レビュー
- 特にLayer 2, Layer 3 の観点で回帰バグを検出する

### 4. ループ上限なし（ユーザー制御）

- バグ0 または ユーザー停止指示まで継続
- 各ラウンド後にユーザーに継続確認
- 過度な最適化ループに陥らないよう、LOWのみ残った場合は終了を提案

### 5. 言語非依存の限界

- 言語固有の慣習（Python PEP 8, JS ESLint等）はサブエージェント側の知識に依存
- 特殊な言語（COBOL, Fortran等）の場合は精度が下がる可能性を自己申告
